# مراجعة خطة العمل - رأي الخبير (Oracle APEX 20+ Years)

بعد مراجعة شاملة لخطة التنفيذ، وجدت أن الأساس **جيد وقوي**، لكن هناك **14 نقطة جوهرية** يجب معالجتها قبل البدء في التنفيذ لضمان أن النظام يكون **Production-Ready** وليس مجرد Prototype.

---

## 🔴 مشاكل حرجة (Critical Issues)

### 1. غياب جدول `tts_departments` (الأقسام)
أنت ذكرت أن المدير يتابع مهام **فريقه**، لكن التصميم الحالي يربط الموظف بالمدير مباشرة فقط عبر `manager_id`. هذا يعمل في الهياكل البسيطة لكنه **لا يكفي** عندما:
- مدير يغادر ويحل محله مدير آخر (يجب تحديث كل الموظفين يدوياً).
- تريد تقارير على مستوى القسم أو الإدارة.

> [!IMPORTANT]
> **التوصية:** إضافة جدول `tts_departments` وربط المستخدمين به، مع الإبقاء على `manager_id` كعلاقة مباشرة للتسلسل الإداري:
> ```sql
> CREATE TABLE tts_departments (
>     dept_id      NUMBER GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
>     dept_name    VARCHAR2(100) UNIQUE NOT NULL,
>     dept_head_id NUMBER,
>     is_active    CHAR(1) DEFAULT 'Y' NOT NULL,
>     created_at   TIMESTAMP DEFAULT SYSTIMESTAMP NOT NULL,
>     CONSTRAINT fk_dept_head FOREIGN KEY (dept_head_id) REFERENCES tts_users(user_id)
> );
> -- ثم إضافة dept_id في tts_users
> ALTER TABLE tts_users ADD dept_id NUMBER REFERENCES tts_departments(dept_id);
> ```

### 2. `approval_status` الافتراضي `PENDING` غير منطقي
حالياً كل مهمة تُنشأ وحالة الاعتماد `PENDING`. هذا خطأ لأن المهمة لم تُكتمل أصلاً لتحتاج اعتماداً.

> [!CAUTION]
> **التوصية:** يجب أن يكون الافتراضي `NOT_SUBMITTED` أو `NULL`، وتتحول إلى `PENDING` فقط عندما يطلب الموظف اعتماد المهمة بعد إنجازها:
> ```sql
> approval_status VARCHAR2(20) DEFAULT 'NOT_SUBMITTED' NOT NULL
>     CHECK (approval_status IN ('NOT_SUBMITTED', 'PENDING', 'APPROVED', 'REJECTED')),
> ```

### 3. غياب جدول `tts_notifications` للإشعارات
ذكرنا أننا قد نحتاج لإشعارات بريدية، لكن لا يوجد جدول لتتبع الإشعارات. بدون هذا الجدول:
- لا يمكن معرفة هل تم إرسال الإشعار بنجاح أم لا.
- لا يمكن عرض إشعارات داخل التطبيق (In-App Notifications).

> [!IMPORTANT]
> **التوصية:** إضافة جدول:
> ```sql
> CREATE TABLE tts_notifications (
>     notification_id  NUMBER GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
>     user_id          NUMBER NOT NULL,
>     task_id          NUMBER,
>     notification_type VARCHAR2(50) NOT NULL, -- 'TASK_ASSIGNED', 'STATUS_CHANGED', 'APPROVAL_REQUIRED', etc.
>     message          VARCHAR2(4000) NOT NULL,
>     is_read          CHAR(1) DEFAULT 'N' NOT NULL CHECK (is_read IN ('Y', 'N')),
>     email_sent       CHAR(1) DEFAULT 'N' NOT NULL CHECK (email_sent IN ('Y', 'N')),
>     created_at       TIMESTAMP DEFAULT SYSTIMESTAMP NOT NULL,
>     CONSTRAINT fk_notif_user FOREIGN KEY (user_id) REFERENCES tts_users(user_id),
>     CONSTRAINT fk_notif_task FOREIGN KEY (task_id) REFERENCES tts_tasks(task_id)
> );
> ```

### 4. Password Hashing: يجب استخدام Salt
الخطة تستخدم `DBMS_CRYPTO` مع SHA-256 فقط. هذا **غير كافٍ** أمنياً لأنه عرضة لهجمات Rainbow Table.

> [!CAUTION]
> **التوصية:** يجب إضافة عمود `password_salt` في `tts_users` واستخدامه مع التشفير:
> ```sql
> ALTER TABLE tts_users ADD password_salt VARCHAR2(128);
> ```
> ثم في الـ Package:
> ```sql
> FUNCTION hash_password(p_password IN VARCHAR2, p_salt IN VARCHAR2) RETURN VARCHAR2 IS
> BEGIN
>     RETURN RAWTOHEX(
>         DBMS_CRYPTO.HASH(
>             UTL_RAW.CAST_TO_RAW(p_salt || p_password),
>             DBMS_CRYPTO.HASH_SH256
>         )
>     );
> END;
> ```

## 🟡 مشاكل مهمة (Important Improvements)

### 5. غياب Indexes على الأعمدة المستخدمة في البحث والفلترة
الجداول مُعرّفة بدون **Indexes** صريحة. في بيئة إنتاج ستلاحظ بطء شديد خاصة في:
- `tts_tasks.assigned_to` (يُستخدم في كل Query للصلاحيات)
- `tts_tasks.status` (يُستخدم في Kanban و Dashboard)
- `tts_tasks.system_id` (يُستخدم في الفلترة)
- `tts_task_history.task_id` (يُستخدم في عرض تاريخ المهمة)
- `tts_comments.task_id`

> **التوصية:** إضافة Indexes صريحة:
> ```sql
> CREATE INDEX idx_tasks_assigned   ON tts_tasks(assigned_to);
> CREATE INDEX idx_tasks_status     ON tts_tasks(status);
> CREATE INDEX idx_tasks_system     ON tts_tasks(system_id);
> CREATE INDEX idx_tasks_created_by ON tts_tasks(created_by);
> CREATE INDEX idx_tasks_dates      ON tts_tasks(start_date, due_date);
> CREATE INDEX idx_history_task     ON tts_task_history(task_id);
> CREATE INDEX idx_comments_task    ON tts_comments(task_id);
> CREATE INDEX idx_users_manager    ON tts_users(manager_id);
> ```

### 6. غياب `tts_task_tags` (العلامات/الوسوم)
للمهام اليومية، القدرة على وضع **Tags** (مثل: `urgent`, `bug`, `enhancement`, `meeting`) مفيدة جداً للفلترة والبحث السريع.

> **التوصية:** إضافة جدول Many-to-Many:
> ```sql
> CREATE TABLE tts_tags (
>     tag_id    NUMBER GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
>     tag_name  VARCHAR2(50) UNIQUE NOT NULL
> );
> CREATE TABLE tts_task_tags (
>     task_id   NUMBER NOT NULL REFERENCES tts_tasks(task_id) ON DELETE CASCADE,
>     tag_id    NUMBER NOT NULL REFERENCES tts_tags(tag_id) ON DELETE CASCADE,
>     PRIMARY KEY (task_id, tag_id)
> );
> ```

### 7. غياب `approval_notes` في جدول المهام

> **التوصية:** إضافة عمود:
> ```sql
> ALTER TABLE tts_tasks ADD approval_notes VARCHAR2(4000);
> ```

## 🟢 تحسينات مقترحة (Nice to Have)

### 9. إضافة `tts_daily_log` لتسجيل ساعات العمل اليومية
حالياً `actual_hours` رقم واحد في المهمة. لكن في الواقع، الموظف يعمل على المهمة **عبر عدة أيام**. بدون جدول تفصيلي لن تستطيع معرفة:
- كم ساعة عمل في كل يوم؟
- ما هو التوزيع الزمني للجهد؟

> **التوصية:**
> ```sql
> CREATE TABLE tts_daily_log (
>     log_id       NUMBER GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
>     task_id      NUMBER NOT NULL REFERENCES tts_tasks(task_id),
>     user_id      NUMBER NOT NULL REFERENCES tts_users(user_id),
>     log_date     DATE NOT NULL,
>     hours_spent  NUMBER(4,2) NOT NULL CHECK (hours_spent > 0 AND hours_spent <= 24),
>     notes        VARCHAR2(4000),
>     created_at   TIMESTAMP DEFAULT SYSTIMESTAMP NOT NULL,
>     CONSTRAINT uk_daily_log UNIQUE (task_id, user_id, log_date)
> );
> ```
> ثم `actual_hours` في `tts_tasks` يتم حسابها تلقائياً من مجموع الساعات في هذا الجدول عبر Trigger أو View.

### 12. إضافة Lookup Tables للقيم الثابتة
حالياً القيم مثل `status` و `priority` محفوظة كـ CHECK Constraints مباشرة. هذا يعمل لكنه **غير مرن** - أي تعديل يحتاج ALTER TABLE.

> **التوصية:** إنشاء جدول Lookup عام:
> ```sql
> CREATE TABLE tts_lookups (
>     lookup_id    NUMBER GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
>     lookup_type  VARCHAR2(50) NOT NULL,  -- 'TASK_STATUS', 'PRIORITY', 'APPROVAL_STATUS'
>     lookup_code  VARCHAR2(50) NOT NULL,
>     display_name VARCHAR2(100) NOT NULL,
>     display_order NUMBER DEFAULT 0,
>     is_active    CHAR(1) DEFAULT 'Y',
>     CONSTRAINT uk_lookups UNIQUE (lookup_type, lookup_code)
> );
> ```

### 14. خطة الـ APEX Navigation محتاجة تفصيل
الخطة لم تذكر:
- **Navigation Menu Structure**: الشريط الجانبي وترتيب العناصر.
- **Breadcrumbs**: مسار التنقل.
- **Authorization Schemes**: كيف سنُعرّف الصلاحيات في APEX (هل Role-based عبر `APEX_UTIL` أم عبر Custom Function؟).

- **Application Theme**: هل سنستخدم Universal Theme مع تخصيصات أم Theme مخصص؟
